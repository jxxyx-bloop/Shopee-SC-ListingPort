A low-risk bridge solution reducing listing duplication from hours to minutes — built and deployed in days, not quarters. Designed by product, ready for engineering to evolve.

The Challenge

A Strategic Seller Segment, Blocked by Operational Friction

Japanese Cross-Border merchants create accounts in each market's local Seller Centre (MY, SG, TH, PH) — no dedicated JP Seller Centre exists today.

Their listings are already in English and ready to cross-post. But today, sellers manually re-enter every product field per market — names, descriptions, prices, images, variations, shipping info.

This is error-prone, slow, and a scaling bottleneck for a high-value segment. A Global Seller Centre has low ROI for this niche. The situation calls for a practical bridge that works today.

Hours → Minutes Time to duplicate a product batch across markets
~500 listings Processed per single transfer session
5 files → 1 file Complexity absorbed by the tool

Decision Framework

Why This Approach, Not the Others

Four options evaluated across feasibility, maintenance cost, time-to-deploy, and evolutionary potential.

Option A: Status Quo

Manual copy-paste, field by field

Pros

  • No dev effort
  • No risk

Cons

  • Hours per batch, high error rate
  • Scales with headcount
  • Seller churn risk
Not scalable — this is the problem

Option B: Browser Automation

Selenium / UiPath

Pros

  • No API dependency
  • End-to-end in theory

Cons

  • UI changes break it immediately
  • Fragile session handling
  • High maintenance burden
  • ToS risk
High cost, fragile, risky

Option C: Open API

Pull listings via API, push to target

Pros

  • Most elegant, fully automated
  • Real-time, no file handling

Cons

  • Internal API access not available today
  • OAuth per seller, longer to deploy
Ideal long-term — blocked today

Option D: File-Based ✓

Leverage existing Mass Upload/Download

Pros

  • Works within existing SC capabilities
  • Zero new infrastructure or permissions
  • Deployable in days
  • Extends to new markets via config

Cons

  • Manual batch trigger (not real-time)
  • Category IDs mapped per market pair
Right tool for right now — chosen

Delivered Solution

What Was Built

Phase 1 — CLI Tool

Python Command-Line Engine

Audience: Technical ops staff, power users

  • Reads 5 Mass Update export files from source market
  • Merges product data by Product ID, handles multi-variation products
  • Converts prices via configurable exchange rates
  • Maps category IDs via JSON config
  • Outputs valid, upload-ready .xlsx for the target market
  • Handles Shopee's non-standard Excel format (solved at engine level)

✓ Production-ready · 60+ automated tests

Phase 2 — Web App

Streamlit Visual Interface

Audience: Non-technical ops staff, seller managers

  • Drag-and-drop upload with auto-detection of export type
  • Visual product preview table before transformation
  • Interactive category mapping UI — no JSON editing needed
  • Exchange rate override field
  • One-click transform and instant file download

✓ Production-ready · Deployable on any laptop or internal server

Note: The web app UI can also be exposed directly to sellers — this is not strictly an internal-only tool.

Evolution Path

Future State — With Shopee Open API

Current (File-Based)

  • Seller downloads 5 export files from source SC
  • Uploads files to web app
  • Reviews product preview and maps categories
  • Downloads transformed output file
  • Uploads to target market SC

Works today. No new permissions needed.

Future (API-Connected)

  • Tool pulls listings directly via API — no file download
  • AI-powered category matching across markets

In Summary

Immediate Value, Designed to Evolve

This tool delivers measurable seller value today using Shopee's existing infrastructure — no new API access, no new permissions, no new infrastructure required.

It is designed to evolve: the file-based approach is a deliberate stepping stone. Every component maps cleanly to an API-driven future. The architecture is already ready for developers to build on.

The web app layer is not limited to internal ops — it can be surfaced directly to sellers, making this a product capability, not just a tooling workaround.
